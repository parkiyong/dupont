<objective>
## Goal
Persist user preferences (Bing market, Spotlight locale, active source) to disk so they survive app restarts.

## Purpose
Users who configure their preferred Bing market or Spotlight locale expect those settings to persist. Currently all preferences reset to defaults on every launch. This is the last functional gap before the app is production-ready.

## Output
- New `app/src/config.rs` module with save/load functions
- App model reads config on startup, writes on settings change
- Config file at `~/.config/damask/config.json`
</objective>


<task type="auto">
  <name>Create config module with save/load</name>
  <files>app/src/config.rs</files>
  <action>
    Create `app/src/config.rs` with:
    - A `Config` struct (derive Serialize, Deserialize) with fields:
      - `bing_market: String` (default "en-US")
      - `spotlight_locale: String` (default "en-US")
      - `active_source: String` (default "bing")
    - A `Config::default()` impl returning sensible defaults
    - A `Config::load() -> Self` function that:
      - Gets config dir via `dirs::config_dir()` / "damask" / "config.json"
      - If file exists, reads and deserializes with serde_json
      - If missing or corrupt, returns defaults
      - NEVER panics on I/O or parse errors
    - A `Config::save(&self) -> anyhow::Result<()>` function that:
      - Creates parent directories if missing (`std::fs::create_dir_all`)
      - Serializes and writes to config path
      - Uses pretty formatting for human readability
    - A `pub fn config_path() -> PathBuf` helper for the config file location

    Use `serde` and `serde_json` from workspace deps (already in root Cargo.toml).
    Use `dirs` crate from workspace deps for XDG config dir.
    Add `serde` and `dirs` to app/Cargo.toml if not already present.
  </action>
  <verify>`cargo check -p damask-app` compiles with the new module</verify>
  <done>AC-1 satisfied: Config struct with load/save exists and handles missing files</done>
</task>

<task type="auto">
  <name>Integrate config into app lifecycle</name>
  <files>app/src/app.rs, app/src/main.rs</files>
  <action>
    Wire config into the app:

    1. In `main.rs`: add `mod config;`

    2. In `App::init()` in `app.rs`:
       - Call `config::Config::load()` at the start
       - Use loaded config values for model initialization:
         - `bing_market` from config instead of hardcoded "en-US"
         - `spotlight_locale` from config instead of hardcoded "en-US"
         - `active_source_id` from config instead of hardcoded "bing"
       - Set `source_dropdown.set_selected()` to match loaded active_source

    3. In `update_with_view` handler for `AppMsg::SettingsChanged`:
       - After updating in-memory values, call `config.save()` with current values
       - Ignore save errors silently (log with eprintln but don't crash)

    4. In `update_with_view` handler for `AppMsg::SourceChanged`:
       - After updating active_source_id, save config to persist source selection

    Avoid: Changing the settings window UI. Avoid: Adding async for config I/O (it's fast, sync is fine).
  </action>
  <verify>
    `cargo build -p damask-app` compiles. Manual test: change settings, restart app, settings retained.
  </verify>
  <done>AC-2 satisfied: Settings persist across app restarts. AC-3 satisfied: Corrupt/missing config handled gracefully.</done>
</task>
